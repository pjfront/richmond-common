# S9.5 Batch Rescan Cleanup — Design Spec

**Date:** 2026-03-11
**Sprint:** S9 — Scanner v3: Signal Architecture
**Status:** Approved design, pending implementation

## Overview

Three pre-rescan cleanups followed by batch rescan and confidence distribution validation. Sequential execution: D1 → R1/I1 → I5 → rescan → V1.

Batch performance optimizations (O1-O5) already shipped (33x speedup, ~7 min runtime). This spec covers the remaining cleanup work and the rescan/validation process.

## D1: Remove Dual Temporal Correlation Path

### Problem

`cloud_pipeline.py` Step 5b (lines 491-521) calls the legacy `scan_temporal_correlations()` directly in retrospective mode. But `scan_meeting_json()` (called at line 423) already runs the v3 `signal_temporal_correlation()` detector internally. Temporal flags get created twice during retrospective rescans.

### Changes

1. **Delete Step 5b block** in `cloud_pipeline.py` (lines 491-521) — the `if scan_mode == "retrospective": ... scan_temporal_correlations(...)` block and its associated flag-saving logic.
2. **Remove the import** of `scan_temporal_correlations` from `cloud_pipeline.py`.
3. **Keep the legacy function** `scan_temporal_correlations()` in `conflict_scanner.py` — it's the CLI mode fallback. Removing it is a separate cleanup (not in scope).
4. **Update tests** in `test_temporal_correlation.py` — `test_cloud_pipeline_retrospective_includes_temporal` must positively assert that `signal_temporal_correlation` (v3 detector) is invoked via `scan_meeting_json`, not just check for the word "temporal" in output. The current test passes trivially because "temporal" appears in comments and log keys. Replace with a mock/spy on `signal_temporal_correlation` that verifies it was called during retrospective mode.

### What doesn't change

The v3 `signal_temporal_correlation()` detector, DB mode processing, composite confidence calculation, frontend behavior.

### Files touched

- `src/cloud_pipeline.py` — delete Step 5b block, remove import
- `tests/test_temporal_correlation.py` — update retrospective test

## R1/I1: Gazetteer-Based Vendor Matching

### Problem

`signal_donor_vendor_expenditure()` uses a two-step approach: (1) extract entity names from agenda text via pattern matching (`extract_entity_names()`), (2) match extracted entities against vendors. This is lossy — extraction patterns miss vendors, and extracted phrases are often too long to match against donor names.

### New approach

Flip the direction. Load all known vendors as a gazetteer, check each vendor name directly against item text using `name_in_text()` (contiguous phrase matching). Same approach `signal_campaign_contribution()` already uses for donor names.

### Changes

1. **Vendor profiling script** — `src/profile_vendors.py` (or CLI flag). Queries `city_expenditures` for `normalized_vendor` stats: count, null rate, length distribution, sample values. Output to stdout. Quick diagnostic to confirm data quality before switching.

2. **Refactor `signal_donor_vendor_expenditure()`** — accept a vendor gazetteer (list of vendor name strings) as a new parameter:
   - For each vendor in the gazetteer, call `cached_name_in_text(vendor, item_text)` using `ctx.nit_cache` as the cache dict (same pattern as `signal_campaign_contribution`)
   - On match, proceed to Stage 2 (vendor-to-donor cross-reference) unchanged
   - **Update call-site guard** in `scan_meeting_json()`: change `if expenditures and entities:` to `if expenditures and vendor_gazetteer:` — the guard must check for the gazetteer, not entities, or vendor signals will be silently suppressed whenever entity extraction returns nothing
   - **Update early-return guard** inside the function: remove the `not entities` check from the `if not entities or ...: return signals` guard at the top of the function
   - The `entities` parameter can be removed from this function's signature (it is not shared with other detectors; each detector has its own signature)

3. **Pre-load vendor gazetteer in DB mode** — in `scan_meeting_json()`, query distinct `normalized_vendor` values from `city_expenditures` once at scan start, pass to the detector. Same pattern as `post_vote_contributions` pre-loading (lines 2157-2173).

4. **Update tests** — `TestSignalDonorVendorExpenditure` passes a vendor gazetteer instead of relying on entity extraction. Add test cases for `name_in_text` phrase matching (vendor found/not found in item text).

### What doesn't change

`extract_entity_names()` itself (other code may use it), Stage 2 donor cross-reference logic, composite confidence, signal output schema.

### Risk mitigation

Vendor profiling script runs first. If data is too dirty (high null rate, names too short for `name_in_text`'s 10-char minimum), we know before committing to the approach.

### Files touched

- `src/profile_vendors.py` — new, vendor data quality profiler
- `src/conflict_scanner.py` — refactor `signal_donor_vendor_expenditure()`, add gazetteer param; update `scan_meeting_json()` to pre-load vendors
- `tests/test_temporal_vendor_signals.py` — update vendor-donor tests

## I5: CAL-ACCESS EXPN_CD Independent Expenditures

### Problem

CAL-ACCESS has 370MB of independent expenditure data (EXPN_CD) connecting PAC money to specific candidates. Chevron's "Coalition for Richmond's Future" spent $3.1M in 2014. This data is documented but not parsed.

### Changes

1. **New function `get_richmond_expenditures()`** in `calaccess_client.py`:
   - Follows `get_richmond_contributions()` pattern exactly
   - Opens `CalAccess/DATA/EXPN_CD.TSV` from bulk ZIP
   - Filters by `filing_map` (reuses `find_richmond_filing_ids()`)
   - Returns list[dict]: filer_id, committee, filing_id, candidate_name, support_or_oppose, amount, date, expenditure_description, city_fips
   - **Column name lookup required:** CAL-ACCESS uses non-obvious names (e.g., `CTRIB_NAML` for contributor last name). Implementer must look up EXPN_CD column names from CAL-ACCESS official documentation at https://www.sos.ca.gov/campaign-lobbying/cal-access-resources/raw-data-campaign-finance-and-lobbying-activity or from the `CalAccess/DATA/EXPN_CD.TSV` header row. Expected fields: `CAND_NAML`/`CAND_NAMF` (candidate name), `SUP_OPP_CD` (support/oppose), `EXPN_DSCR` (description), `AMOUNT` (amount), `EXPN_DATE` (date). Defensive `.get()` access throughout.

2. **New `independent_expenditures` table** (migration 029):
   - Different entity shape than contributions: connects committee → candidate with support/oppose flag
   - Columns: id, city_fips, committee_name (VARCHAR, not FK -- matches contributions pattern where committee is stored as string), candidate_name, support_or_oppose, amount, expenditure_date, description, filing_id, source, created_at
   - Index on city_fips, candidate_name
   - Note: existing contributions flow stores committee as a string via `db.load_contributions_to_db()`. Follow the same pattern here. Committee normalization/FK wiring is a separate future step.

3. **DB loading function** — `load_expenditures_to_db()` in `db.py`:
   - Stores committee_name as string (same pattern as contributions)
   - Inserts into `independent_expenditures`
   - Tracks source counts in scan_runs JSONB

4. **Tests** — all mocked (no ZIP needed):
   - `get_richmond_expenditures()` with mock ZIP containing sample EXPN_CD rows
   - DB loading
   - Column name defensiveness (missing/extra columns don't crash)

5. **CLI entry point** — `--expn` flag or subcommand for running the parser

### Scanner integration (deferred)

How IE data affects conflict scores ships separately. Parser + DB loading land now. Signal integration can ride with the batch rescan or become a follow-on.

### What doesn't change

`get_richmond_contributions()`, `find_richmond_filing_ids()` (reused), existing contribution signals, bulk download process.

### Files touched

- `src/calaccess_client.py` — new `get_richmond_expenditures()`
- `src/db.py` — new `load_expenditures_to_db()`
- `src/migrations/029_independent_expenditures.sql` — new table
- `tests/test_calaccess_client.py` — new expenditure tests

## Batch Rescan + V1 Validation

### Rescan

After D1 + R1/I1 + I5 land, run the scanner in batch mode against all meetings. Human-triggered. Expected ~7 min runtime (O1-O5 optimizations).

Note: I5's EXPN_CD data won't be in this rescan (bulk ZIP not downloaded). Rescan validates D1 (no double-counted temporal flags) and R1/I1 (improved vendor matching). EXPN_CD integration rescans separately later.

### V1 Validation

Python script that queries scan results and produces a comparison report:

- Total flag count (before vs after)
- Percentage of flags above 0.50 (public visibility threshold)
- Distribution by signal type (temporal, campaign, donor-vendor, form700)
- Flags that appeared/disappeared/changed tier
- Spot-check prompt: highest-confidence flags for manual review

Output: stdout or markdown. One-time diagnostic, not a permanent feature.

### Files touched

- `src/validate_rescan.py` — new, V1 comparison script
